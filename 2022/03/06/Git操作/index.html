git pull=git fetch + git merge

现在来看看git merge和git rebase的区别。

如果有3次提交A，B，C。

在远程分支origin的基础上创建一个名叫*”mywork”*的分支并且提交了，同时如果有其他人在origin上做了一些修改并且提交了，这个时候其实不应该提交”mywork”分支，因为提交后会发生冲突，那我们如何解决呢？一般有两种方式解决：

  • git merge

    用git pull命令将”origin”分支上的修改pull下来与本地提交合并(merge)成版本M，但是这样会形成一个菱形的提交图形。

  • git rebase

    创建一个新的提交信息R，R的文件内容和M是一致的，但是我们已经将之前的mywork那次提交废除掉了。由于这种删除，避免了菱形的产生，保持提交曲线为直线，会更好看更好理解。

在任何时候都可以用==git rebase –abort== 参数来终止rebase的行动。

常见场景
分支合并
  • 需要另一个分支的所有代码进行更新变动

这种情况需要使用git merge命令进行两个分支的合并操作。

  • 只需要部分代码变动

需要使用==cherry-pick==命令。

cherry-pick命令的作用就是将指定的提交（commit）应用于其他分支。

此命令会将指定的提交应用于当前分支，这会在当前分支产生一个新的提交。

cherry-pick的配置项

配置项 配置项说明
-e, –edit 打开外部编辑器，编辑提交信息
-n, –no-commit 只更新工作区和暂存区，不产生新的提交
-x 在提交信息的末尾追加一行cherry picked from commit… ，方便以后查到这个提交是如何产生的
-s, –signoff 在提交信息的末尾追加一行操作者的签名，表示是谁进行了这个操作
-m parent-number,–mainline parent-number 如果原始提交是一个合并节点，来自于两个分支的合并，那么cherry-pick默认将失败，因为他不知道应该采用那个分支的代码变动。
